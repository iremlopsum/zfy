# Test Coverage Goals

This document outlines the test coverage goals and current status for the zfy library.

## Coverage Targets

### Overall Coverage Goals
- **Statements**: ≥ 95%
- **Branches**: ≥ 90%
- **Functions**: ≥ 95%
- **Lines**: ≥ 95%

### Critical Paths (100% Coverage Required)
- `src/core/create-store.ts` - Store creation logic
- `src/core/init-stores.ts` - Multi-store initialization
- `src/internals/validations.ts` - Input validation
- `src/internals/invariant.ts` - Error handling

### Component Coverage (≥ 90%)
- `src/core/PersistGate.tsx` - Rehydration gate component
- `src/core/use-rehydrate.ts` - Rehydration hook

### Middleware Coverage (≥ 95%)
- `src/internals/middlewares/create-middlewares.ts` - Middleware composition
- `src/internals/middlewares/logger-middleware.ts` - Logger
- `src/internals/middlewares/persist-middleware.ts` - Persistence
- `src/internals/middlewares/subscribe-middleware.ts` - Subscriptions

## Running Coverage Reports

### Generate HTML Coverage Report
```bash
yarn test --coverage
```

### View Coverage in Browser
```bash
open coverage/lcov-report/index.html
```

### Generate Coverage Summary
```bash
yarn test --coverage --coverageReporters=text-summary
```

### Generate JSON Coverage Data
```bash
yarn test --coverage --coverageReporters=json
```

## Coverage Configuration

Coverage is configured in `package.json` under the `jest` section:

```json
{
  "collectCoverageFrom": [
    "src/**/*.{ts,tsx}",
    "!src/**/*.test.{ts,tsx}",
    "!src/**/__tests__/**",
    "!src/**/index.ts"
  ],
  "coverageThreshold": {
    "global": {
      "statements": 95,
      "branches": 90,
      "functions": 95,
      "lines": 95
    }
  }
}
```

## Analyzing Coverage Gaps

### Find Untested Lines
1. Run `yarn test --coverage`
2. Open `coverage/lcov-report/index.html`
3. Navigate to files with < 100% coverage
4. Red/pink highlighted code = untested
5. Add tests for uncovered paths

### Common Coverage Gaps
- Error handling paths
- Edge case validations
- Async error scenarios
- Type guard branches
- Optional parameter paths

## Coverage Best Practices

### 1. Test Behavior, Not Implementation
Focus on testing what the code does, not how it does it.

### 2. Cover All Branches
Every if/else, switch case, and ternary should have tests.

### 3. Test Error Paths
Don't just test the happy path - test failures too.

### 4. Use Mutation Testing
Consider tools like Stryker to verify test quality.

### 5. Avoid Coverage Theater
100% coverage doesn't mean bug-free code. Write meaningful tests.

## Coverage Reports in CI/CD


## Coverage Exemptions

Some code paths may be intentionally untested:

### Type-Only Code
- Type definitions without runtime logic
- Type guards that are compile-time only

### Development-Only Code
- Console logging in development mode
- Debug utilities

### External Dependencies
- Third-party library integration points
- Platform-specific code that can't be tested in JSDOM

## Monitoring Coverage Over Time

### Track Coverage Trends
- Set up Codecov or Coveralls
- Monitor coverage on every PR
- Prevent coverage regressions

### Coverage Requirements for PRs
- New features must include tests
- Bug fixes should add regression tests
- Refactors should maintain or improve coverage
